## 1. 正常系テストケース（基本的な動作）

### 1.1 マイグレーション管理テーブル作成

- **テスト名**: schema_migrationsテーブル初期化テスト
  - **何をテストするか**: マイグレーション管理テーブルが正しく作成されること
  - **期待される動作**: schema_migrationsテーブルが適切なスキーマで作成される
- **入力値**: 新しいDatabase instance
  - **入力データの意味**: まっさらなデータベース状態からの初期化
- **期待される結果**:
  ```sql
  CREATE TABLE schema_migrations (
      version INTEGER PRIMARY KEY,
      applied_at TEXT NOT NULL,
      checksum TEXT NOT NULL
  )
  ```
  - **期待結果の理由**: マイグレーション履歴追跡に必要最小限の構造
- **テストの目的**: マイグレーションシステムの基盤となるテーブルが正常に作成される
  - **確認ポイント**: テーブル存在確認、スキーマ構造確認
- 🔵 **青信号** - 要件定義書の実装方針に基づく

### 1.2 マイグレーションファイル読み込み

- **テスト名**: 既存マイグレーションファイル読み込みテスト
  - **何をテストするか**: src-tauri/migrations/ディレクトリからファイルが正しく読み込まれる
  - **期待される動作**: 001, 002番のマイグレーションファイルが正しく解析される
- **入力値**: `src-tauri/migrations/`ディレクトリパス
  - **入力データの意味**: 実際のマイグレーションファイルが配置されているディレクトリ
- **期待される結果**:
  ```rust
  vec![
      Migration { version: 1, description: "Initial player note schema", up_sql: "...", down_sql: "..." },
      Migration { version: 2, description: "Add performance indexes", up_sql: "...", down_sql: "..." }
  ]
  ```
  - **期待結果の理由**: 既存の2つのマイグレーションファイルが正確に読み込まれる
- **テストの目的**: ファイルシステムからマイグレーション定義を正しく読み込める
  - **確認ポイント**: ファイル順序、SQL内容、メタデータ解析
- 🔵 **青信号** - 既存マイグレーションファイルの存在確認済み

### 1.3 最新バージョンまでのマイグレーション実行

- **テスト名**: 最新バージョンマイグレーション実行テスト
  - **何をテストするか**: migrate_to_latest()で全マイグレーションが順次実行される
  - **期待される動作**: バージョン0→2まで順次適用され、全テーブルが作成される
- **入力値**: 空のデータベース + `migrate_to_latest()`実行
  - **入力データの意味**: 初期状態からの完全なマイグレーション実行
- **期待される結果**:
  ```rust
  MigrationResult {
      success: true,
      applied_migrations: vec![1, 2],
      current_version: 2,
      error_message: None
  }
  ```
  - **期待結果の理由**: 2つのマイグレーションが成功し、最新バージョンに到達
- **テストの目的**: エンドツーエンドのマイグレーション実行が正常に完了する
  - **確認ポイント**: テーブル作成確認、インデックス作成確認、履歴記録確認
- 🔵 **青信号** - 要件定義書のデータフローに基づく

### 1.4 特定バージョンへのマイグレーション

- **テスト名**: 特定バージョン指定マイグレーションテスト
  - **何をテストするか**: migrate_to_version(1)でバージョン1まで適用される
  - **期待される動作**: バージョン1のマイグレーションのみ実行される

  - **期待結果の理由**: 指定したバージョンまでのみ適用される
- **テストの目的**: バージョン制御が正確に動作する
  - **確認ポイント**: バージョン2のインデックスが作成されていないこと
- 🔵 **青信号** - 要件定義書の基本使用パターンに記載

### 1.5 マイグレーション状態確認

- **テスト名**: マイグレーション状態取得テスト
  - **何をテストするか**: get_migration_status()で現在の状態が正確に取得される
  - **期待される動作**: 適用済み・未適用マイグレーションが正しく分類される
- **入力値**: バージョン1まで適用済みのデータベース

- **期待される結果**:

  - **期待結果の理由**: 現在の状態が正確に反映される
- **テストの目的**: マイグレーション管理情報の正確な取得
  - **確認ポイント**: 適用日時、チェックサム、未適用の特定
- 🔵 **青信号** - 要件定義書の出力仕様に基づく

## 2. 異常系テストケース（エラーハンドリング）

- **テスト名**: 無効SQLマイグレーション失敗テスト
  - **エラーケースの概要**: 構文エラーのあるSQLでマイグレーションが失敗する


  - **システムの安全性**: トランザクションロールバックで元の状態を維持

  - **品質保証の観点**: データ破損防止、デバッグ容易性確保

  - **エラーメッセージの内容**: "Migration directory not found: /path/to/migrations"
  - **システムの安全性**: 不完全な状態でのマイグレーション実行を回避
- **テストの目的**: ファイルアクセスエラーの検出
  - **品質保証の観点**: 運用時のトラブル早期発見
- 🔵 **青信号** - 要件定義書のエラーケースに記載

### 2.3 チェックサム不一致エラー

- **テスト名**: マイグレーションファイル改竄検出テスト
  - **エラーケースの概要**: 適用済みマイグレーションのチェックサムが不一致

- **期待される結果**: チェックサム不一致エラーと実行停止
  - **エラーメッセージの内容**: "Checksum mismatch for migration version 1"
  - **システムの安全性**: データ整合性の保護
- **テストの目的**: マイグレーション改竄の検出
  - **品質保証の観点**: データベース整合性の保証

## テストケース実装時の日本語コメント指針

以下の日本語コメントパターンを使用してテストを実装します：

```rust
#[tokio::test]
async fn test_migration_table_initialization() {
    // 【テスト目的】: マイグレーション管理テーブルが正しく作成されることを確認
    // 【テスト内容】: 新規データベースでMigrator初期化時の動作をテスト
    // 【期待される動作】: schema_migrationsテーブルが適切なスキーマで作成される
    // 🔵 青信号 - 要件定義書の実装方針に基づく

    // 【テストデータ準備】: インメモリデータベースで初期状態を作成
    let db = Database::new_test().expect("Failed to create test database");

    // 【初期条件設定】: マイグレーションシステム未初期化の状態
    let migrator = Migrator::new(&db).expect("Failed to create migrator");

    // 【実際の処理実行】: マイグレーション管理テーブル初期化を実行
    let result = migrator.initialize_migration_table();

    // 【結果検証】: schema_migrationsテーブルの存在と構造を確認
    assert!(result.is_ok()); // 【確認内容】: 初期化処理が成功すること

    let conn = db.0.lock().unwrap();
    let table_exists = conn.prepare("SELECT name FROM sqlite_master WHERE type='table' AND name='schema_migrations'");
    assert!(table_exists.is_ok()); // 【確認内容】: schema_migrationsテーブルが作成されること
}
```
