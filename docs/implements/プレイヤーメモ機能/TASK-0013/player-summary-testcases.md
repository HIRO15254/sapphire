## 1. 正常系テストケース

### TC-UPDATE-SUMMARY-001: 総合メモの初回更新

- **テスト名**: 総合メモの初回更新（空の状態から更新）
  - **何をテストするか**: プレイヤー作成時に自動生成された空の総合メモを初めて編集できることを確認
  - **期待される動作**: HTML内容が正常に更新され、updated_atが自動更新される
  - 🔵 信頼性レベル: 要件定義書（REQ-304: HTML形式保存）に基づく

- **入力値**:
  - `player_id`: 1（事前に作成したプレイヤー）
  - `content`: `"<h1>攻略メモ</h1><p>初心者プレイヤー</p>"`
  - **入力データの意味**: 実際のユーザーが初めて総合メモを編集するシナリオを想定

- **期待される結果**:
  - `Result::Ok(PlayerSummary)` が返される
  - `PlayerSummary.content` が `"<h1>攻略メモ</h1><p>初心者プレイヤー</p>"` と一致
  - `PlayerSummary.player_id` が `1` と一致
  - `PlayerSummary.updated_at` が現在時刻に近い値
  - **期待結果の理由**: REQ-304（HTML形式保存）を満たし、updated_atが自動更新されることを保証

- **テストの目的**: 基本的な更新機能が正常に動作することを確認
  - **確認ポイント**:
    - HTML内容が正しく保存される
    - updated_atが自動的に現在時刻に更新される
    - PlayerSummaryエンティティが正しく返される

### TC-UPDATE-SUMMARY-002: 総合メモの複数回更新

- **テスト名**: 総合メモを2回連続で更新し、最新の内容が保存される
  - **何をテストするか**: 既に内容がある総合メモを上書き更新できることを確認
  - **期待される動作**: 古い内容が新しい内容で置き換わり、updated_atが2回目の更新時刻に更新される
  - 🔵 信頼性レベル: 要件定義書（ケース3: 複数回更新）に基づく

- **入力値**:

  - **2回目**: `player_id=1`, `content="<p>内容2</p>"`

- **期待される結果**:

  - 2回目更新後: `content="<p>内容2</p>"`, `updated_at=T2` (T2 > T1)
  - **期待結果の理由**: 最新の編集内容が保存され、タイムスタンプが更新されることを保証

- **テストの目的**: 上書き更新機能とupdated_at自動更新を確認
  - **確認ポイント**:
    - 古い内容が完全に置き換わる
