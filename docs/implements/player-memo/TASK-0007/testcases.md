## 1. 正常系テストケース（基本的な動作）

### TC-PLAYER-CAT-001: プレイヤーに種別を割り当て（1-to-1関係）

- **テスト名**: プレイヤー作成時に種別を割り当て、正しく関連付けられることを確認
  - **何をテストするか**: プレイヤーと種別の1対1関連付けの基本動作
  - **期待される動作**: プレイヤー作成時に指定した種別IDが正しく保存される
- **入力値**:
  - 種別: name = "タイト", color = "#FF0000"
  - プレイヤー: name = "山田太郎", category_id = 1
  - **入力データの意味**: 実際のユーザーが種別を設定してプレイヤーを登録するシナリオを再現
- **期待される結果**:
  - プレイヤー情報: id = 1, name = "山田太郎", category_id = Some(1)
  - **期待結果の理由**: REQ-104（1プレイヤーに最大1つの種別）を満たすため
- **テストの目的**: 基本的な種別割り当て機能の正常動作を確認
  - **確認ポイント**: プレイヤーのcategory_idが種別IDと一致する
- 🔵 **信頼性レベル**: REQ-104、schema.rs:15（category_id定義）に基づく

---

### TC-PLAYER-CAT-002: 種別を別の種別に変更

- **テスト名**: プレイヤーの種別を別の種別に変更し、正しく更新されることを確認
  - **何をテストするか**: 種別の再割り当て機能
  - **期待される動作**: UPDATE文で種別IDを変更した際、正しく反映される
- **入力値**:
  - 種別A: name = "タイト", color = "#FF0000" (id = 1)
  - 種別B: name = "ルース", color = "#00FF00" (id = 2)
  - プレイヤー: name = "田中一郎", 初期category_id = 1 → 変更後 = 2
  - **入力データの意味**: ユーザーがプレイヤーの分類を変更するシナリオ
- **期待される結果**:
  - プレイヤー情報: category_id = Some(2)
  - **期待結果の理由**: プレイヤーは常に最新の種別情報を保持すべき
- **テストの目的**: 種別の柔軟な変更が可能であることを確認
  - **確認ポイント**: UPDATE後のcategory_idが新しい種別IDに変更されている
- 🔵 **信頼性レベル**: REQ-104、既存パターン（integration_tests.rs:66-97）に基づく

---

### TC-PLAYER-CAT-003: 種別をNULLに設定（種別解除）

- **テスト名**: プレイヤーの種別をNULLに設定し、種別が解除されることを確認
  - **何をテストするか**: 種別の解除機能（category_id = NULL）
  - **期待される動作**: UPDATE文でcategory_id = NULLを設定した際、種別が解除される
- **入力値**:
  - プレイヤー: name = "佐藤花子", 初期category_id = 1 → 変更後 = NULL
  - **入力データの意味**: ユーザーがプレイヤーの種別を「なし」に戻すシナリオ
- **期待される結果**:
  - プレイヤー情報: category_id = None
  - **期待結果の理由**: schema.rs:15でcategory_idはNULL許可のため
- **テストの目的**: 種別の任意性を確認（種別設定が強制ではない）
  - **確認ポイント**: category_idがNoneになっている
- 🔵 **信頼性レベル**: schema.rs:15（NULL許可）、REQ-104に基づく

---

### TC-PLAYER-CAT-004: 種別を編集した際の影響確認（統合テスト）

- **テスト名**: 種別名を編集した際、全プレイヤーから参照される種別情報が更新されることを確認
  - **何をテストするか**: 種別編集時の変更伝播（REQ-502）
  - **期待される動作**: 種別テーブルのnameを更新すると、JOINで取得される種別名も更新される
- **入力値**:
  - 種別: name = "タイト" → "タイトパッシブ"に変更
  - プレイヤー: 3人（A, B, C）すべてcategory_id = 1
  - **入力データの意味**: 複数のプレイヤーが同じ種別を使用している実運用シナリオ
- **期待される結果**:
  - 全プレイヤーのJOIN結果: category.name = "タイトパッシブ"
  - **期待結果の理由**: 種別は参照関係のため、種別テーブルを更新すれば全プレイヤーに反映される
- **テストの目的**: REQ-502（種別編集時の変更伝播）の実装確認
  - **確認ポイント**: 3人すべてのプレイヤーで種別名が更新されている
- 🔵 **信頼性レベル**: REQ-502、schema.rs:18（FOREIGN KEY定義）に基づく

---

### TC-PLAYER-CAT-005: 種別を削除した際のSET NULL動作確認（統合テスト）

- **テスト名**: 種別を削除した際、関連プレイヤーのcategory_idが自動的にNULLに設定されることを確認
  - **何をテストするか**: ON DELETE SET NULL制約の動作（REQ-504, NFR-202）
  - **期待される動作**: 種別を削除すると、該当プレイヤーのcategory_idがNULLになる
- **入力値**:
  - 種別: name = "アグレッシブ", id = 1
  - プレイヤー: 2人（X, Y）category_id = 1
  - 削除: DELETE FROM player_categories WHERE id = 1
  - **入力データの意味**: ユーザーが不要になった種別を削除するシナリオ
- **期待される結果**:
  - プレイヤーX: category_id = None
  - プレイヤーY: category_id = None
  - **期待結果の理由**: schema.rs:18のON DELETE SET NULL制約により自動的にNULLになる
- **テストの目的**: REQ-504, NFR-202（種別削除時の安全な解除）の実装確認
  - **確認ポイント**: 2人のプレイヤーすべてでcategory_idがNoneになっている
- 🔵 **信頼性レベル**: REQ-504, NFR-202、schema.rs:18（ON DELETE SET NULL）に基づく

## 2. 異常系テストケース（エラーハンドリング）

### TC-PLAYER-CAT-ERR-001: 存在しない種別IDを割り当て（外部キー制約違反）

- **テスト名**: 存在しない種別IDでプレイヤーを作成した際、エラーが発生することを確認
  - **エラーケースの概要**: 外部キー制約違反（FOREIGN KEY constraint failed）
  - **エラー処理の重要性**: データの整合性を保つため、無効な参照を防ぐ必要がある
- **入力値**:
  - プレイヤー: name = "無効プレイヤー", category_id = 999（存在しないID）
  - **不正な理由**: category_id = 999の種別が存在しない
  - **実際の発生シナリオ**: フロントエンドのバグや、種別削除後の古いデータ参照
- **期待される結果**:
  - エラー: `FOREIGN KEY constraint failed` または同等のエラー
  - **エラーメッセージの内容**: SQLiteの標準エラーメッセージ
  - **システムの安全性**: データベースが不整合を防ぐため、INSERT/UPDATEを拒否
- **テストの目的**: 外部キー制約が正しく機能することを確認
  - **品質保証の観点**: データベース層で不正なデータを防ぐ
- 🔵 **信頼性レベル**: schema.rs:18（FOREIGN KEY制約）、SQLite仕様に基づく

## 3. 境界値テストケース

### TC-PLAYER-CAT-EDGE-001: プレイヤー作成時に種別なし（NULL）で作成

- **テスト名**: プレイヤー作成時にcategory_id = NULLで作成できることを確認
  - **境界値の意味**: 種別設定が任意（必須ではない）であることの確認
  - **境界値での動作保証**: category_id = NULLが有効な状態として扱われる
- **入力値**:
  - プレイヤー: name = "種別なしプレイヤー", category_id = NULL
  - **境界値選択の根拠**: schema.rs:15でcategory_idはNULL許可
  - **実際の使用場面**: 種別未設定のままプレイヤーを登録するケース
- **期待される結果**:
  - プレイヤー情報: id = 1, name = "種別なしプレイヤー", category_id = None
  - **境界での正確性**: NULL値が正しく保存される
  - **一貫した動作**: 種別ありとなしで同じCRUD操作が可能
- **テストの目的**: 種別なしプレイヤーの作成が可能であることを確認
  - **堅牢性の確認**: NULLを含むデータでもシステムが正常動作する
- 🔵 **信頼性レベル**: schema.rs:15（NULL許可）、REQ-104に基づく

---

### TC-PLAYER-CAT-EDGE-002: 同じ種別に複数プレイヤーを割り当て

- **テスト名**: 同じ種別IDを複数のプレイヤーに割り当てられることを確認
  - **境界値の意味**: 1つの種別が複数プレイヤーから参照される（多対1関係）
  - **境界値での動作保証**: 同じcategory_idを持つプレイヤーが複数存在できる
- **入力値**:
  - 種別: id = 1, name = "タイト"
  - プレイヤー: 3人すべてcategory_id = 1
  - **境界値選択の根拠**: プレイヤー→種別は多対1関係
  - **実際の使用場面**: 同じプレイスタイルのプレイヤーが複数いるケース
- **期待される結果**:
  - 3人のプレイヤーすべてでcategory_id = Some(1)
  - **境界での正確性**: 参照制約違反が発生しない
  - **一貫した動作**: すべてのプレイヤーが種別を参照できる
- **テストの目的**: 多対1関係が正しく機能することを確認
  - **堅牢性の確認**: 複数プレイヤーからの参照でも問題ない
- 🔵 **信頼性レベル**: schema.rs:18（FOREIGN KEY定義）、データベース正規化理論に基づく
