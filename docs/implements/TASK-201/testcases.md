# TDDテストケースの洗い出し

**【機能名】**: TASK-201 ナビゲーション統合とルーティング

## 開発言語・フレームワーク

- **プログラミング言語**: TypeScript (React 19.1.0)
  - **言語選択の理由**: 型安全性によるルーティング設定の検証、NavigationItem型との統合、React Router v6の型サポート活用
  - **テストに適した機能**: 型定義によるルート設定の検証、コンパイル時の型チェック、ナビゲーション状態の型安全性

- **テストフレームワーク**: Vitest + React Testing Library + @testing-library/react-router
  - **フレームワーク選択の理由**: 既存ResponsiveLayoutテストとの一貫性、React Router v6対応、高速実行とモック機能
  - **テスト実行環境**: jsdom + 新規追加のreact-router-dom ^6.x統合環境
- 🟢 この内容の信頼性レベル: package.jsonとResponsiveLayout.test.tsxから確認済み

## 1. 正常系テストケース（基本的な動作）

### TC-001: React Routerの基本統合動作
- **テスト名**: BrowserRouterとNavigationProviderが正しく統合される
  - **何をテストするか**: BrowserRouter + NavigationProvider + ResponsiveLayoutの3層統合動作
  - **期待される動作**: ルーティング機能付きのレスポンシブレイアウトが正常に初期化される
- **入力値**:
  ```typescript
  navigationConfig = {
    primary: [
      { id: '1', label: 'ホーム', path: '/', icon: IconHome },
      { id: '2', label: 'ユーザー', path: '/users', icon: IconUsers }
    ],
    secondary: [
      { id: '3', label: '設定', path: '/settings', icon: IconSettings }
    ]
  }
  initialRoute = '/'  // React Router初期ルート
  ```
  - **入力データの意味**: 実際のSPAで使用される典型的なナビゲーション構成とホームページ初期表示
- **期待される結果**:
  - BrowserRouterコンポーネントが存在する
  - NavigationProviderコンテキストが利用可能である
  - ResponsiveLayoutが正常にレンダリングされる
  - 初期ルート'/'が正しく表示される
  - **期待結果の理由**: REQ-202「現在のページを視覚的に強調」の基盤となるルーティング統合
- **テストの目的**: ルーティング統合の基本動作確認
  - **確認ポイント**: React Router v6とMantine AppShellの競合回避、Context APIの正常動作
- 🟢 TASK-201要件定義とReact Router v6ドキュメントから確認済み

### TC-002: アクティブページ判定とハイライト表示
- **テスト名**: 現在のルートに基づいてナビゲーション項目がアクティブ状態になる
  - **何をテストするか**: useLocation()による現在ルート検知→ナビゲーション項目のアクティブ状態更新
  - **期待される動作**: '/users'にアクセス時、ユーザーナビゲーション項目が視覚的に強調される
- **入力値**:
  ```typescript
  currentRoute = '/users'
  navigationConfig = {
    primary: [
      { id: '1', label: 'ホーム', path: '/', icon: IconHome },
      { id: '2', label: 'ユーザー', path: '/users', icon: IconUsers },
      { id: '3', label: 'ノート', path: '/notes', icon: IconNotes }
    ],
    secondary: []
  }
  ```
  - **入力データの意味**: 複数ページを持つSPAでの現在位置検知とアクティブ状態管理
- **期待される結果**:
  - HeaderNavigationで'ユーザー'項目がアクティブクラス/属性を持つ
  - FooterNavigationで'ユーザー'項目がアクティブクラス/属性を持つ
  - 他のナビゲーション項目はアクティブ状態ではない
  - ARIA属性（aria-current="page"）が適切に設定される
  - **期待結果の理由**: REQ-202「現在のページを視覚的に強調しなければならない」の直接実装
- **テストの目的**: アクティブページ判定ロジックの確認
  - **確認ポイント**: 全ナビゲーションコンポーネントでのアクティブ状態同期、ARIA属性の正確性
- 🟢 REQ-202とWCAG 2.1 AAアクセシビリティガイドラインから確認済み

### TC-003: パンくずリスト自動生成機能
- **テスト名**: 現在のルートに基づいてBreadcrumbが自動生成される
  - **何をテストするか**: ルート階層解析→ナビゲーション構造解析→動的パンくず生成
  - **期待される動作**: 深い階層のページでホームからの階層パスが表示される
- **入力値**:
  ```typescript
  currentRoute = '/users/123/profile'
  navigationConfig = {
    primary: [
      { id: '1', label: 'ホーム', path: '/', icon: IconHome },
      { id: '2', label: 'ユーザー', path: '/users', icon: IconUsers }
    ],
    secondary: []
  }
  routeConfig = [
    { path: '/', breadcrumb: 'ホーム' },
    { path: '/users', breadcrumb: 'ユーザー一覧' },
    { path: '/users/:id/profile', breadcrumb: 'プロフィール' }
  ]
  ```
  - **入力データの意味**: 階層構造を持つSPAでのユーザーの現在位置把握
- **期待される結果**:
  - Breadcrumbコンポーネントが表示される
  - "ホーム > ユーザー一覧 > プロフィール" の階層が表示される
  - 各パンくず項目がクリック可能な状態である
  - 最後の項目（現在ページ）はクリック不可状態である
  - **期待結果の理由**: ユーザビリティ向上とナビゲーション位置把握の重要性
- **テストの目的**: パンくずリスト生成ロジックの確認
  - **確認ポイント**: 階層解析の正確性、動的生成の性能、クリック可能性の制御
- 🟡 一般的なBreadcrumb UXパターンと要件定義から妥当な推測

### TC-004: ナビゲーション状態管理（NavigationProvider）
- **テスト名**: NavigationProviderが全コンポーネント間で状態を正しく共有する
  - **何をテストするか**: Context APIによる中央集権的なナビゲーション状態管理
  - **期待される動作**: どこからでも現在のルート、パンくず、ローディング状態にアクセスできる
- **入力値**:
  ```typescript
  providerState = {
    activeRoute: '/settings',
    breadcrumbs: [
      { label: 'ホーム', path: '/' },
      { label: '設定', path: '/settings' }
    ],
    isLoading: false
  }
  ```
  - **入力データの意味**: NavigationProviderが管理する全状態情報の代表的な状態
- **期待される結果**:
  - useNavigationContext()でactiveRouteが取得できる
  - useNavigationContext()でbreadcrumbsが取得できる
  - useNavigationContext()でisLoadingが取得できる
  - 状態更新時に全コンシューマーコンポーネントが再レンダリングされる
  - **期待結果の理由**: 中央集権的な状態管理によるコンポーネント間の一貫性保証
- **テストの目的**: Context APIの状態管理機能確認
  - **確認ポイント**: Context Providerの範囲、状態更新の伝播、パフォーマンスへの影響
- 🟢 React Context APIドキュメントと要件定義から確認済み

### TC-005: モバイルでのページ遷移後メニュー自動クローズ
- **テスト名**: モバイルレイアウトでナビゲーション後にハンバーガーメニューが自動でクローズされる
  - **何をテストするか**: useNavigate()実行→ルート変更検知→ハンバーガーメニュー状態リセット
  - **期待される動作**: モバイルでのナビゲーション体験向上とメニューの適切な制御
- **入力値**:
  ```typescript
  viewport = { width: 400, height: 800 }  // モバイルサイズ
  hamburgerOpened = true  // メニュー開始状態
  navigationAction = navigate('/users')  // ページ遷移実行
  ```
  - **入力データの意味**: モバイルユーザーがハンバーガーメニューからページ遷移する典型的なフロー
- **期待される結果**:
  - ページ遷移前: ハンバーガーメニューが開いている（opened=true）
  - navigate('/users')実行
  - ページ遷移後: ハンバーガーメニューが自動でクローズされる（opened=false）
  - ページ遷移後: 新しいページが正しく表示される
  - **期待結果の理由**: モバイルUXベストプラクティスとTASK-201の明示的要件
- **テストの目的**: モバイル特有のナビゲーション動作確認
  - **確認ポイント**: useNavigateとuseLocationの連携、メニュー状態管理、タイミング制御
- 🟢 TASK-201要件「モバイル対応: 遷移後のハンバーガーメニュー自動クローズ」から確認済み
