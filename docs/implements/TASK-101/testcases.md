# TDDテストケースの洗い出し

**【機能名】**: ResponsiveLayout（最上位）コンポーネント

## 開発言語・フレームワーク

- **プログラミング言語**: TypeScript (React 19.1.0)
  - **言語選択の理由**: 型安全性によるバグ削減とMantine 8.3.0の型サポート活用
  - **テストに適した機能**: 型定義による入力値の検証、コンパイル時の型チェック

- **テストフレームワーク**: Vitest + React Testing Library + Jest DOM
  - **フレームワーク選択の理由**: 高速な実行速度、React 19対応、Mantine互換性
  - **テスト実行環境**: Node.js (jsdom) + Vite開発環境統合
- 🟢 この内容の信頼性レベル: package.jsonから確認済み

## 1. 正常系テストケース（基本的な動作）

### TC-001: モバイルレイアウト表示の基本動作
- **テスト名**: 768px以下でモバイルレイアウトが正しく表示される
  - **何をテストするか**: useMediaQuery('(max-width: 48em)')でisMobile=trueの際のレイアウト構成
  - **期待される動作**: Header、Footer、Drawerコンポーネントが適切に配置される
- **入力値**:
  ```typescript
  navigationConfig = {
    primary: [{ id: '1', label: 'ホーム', path: '/', icon: IconHome }],
    secondary: [{ id: '2', label: '設定', path: '/settings', icon: IconSettings }]
  }
  viewport = { width: 400, height: 800 }  // モバイルサイズ
  ```
  - **入力データの意味**: 768px以下の一般的なスマートフォン画面サイズを想定
- **期待される結果**:
  - AppShell.Header存在確認
  - AppShell.Footer存在確認（モバイルのみ）
  - AppShell.Navbar非表示確認
  - ハンバーガーメニューボタン表示確認
  - **期待結果の理由**: REQ-101「画面幅が768px以下の場合、スマートフォンレイアウトを適用」に基づく
- **テストの目的**: モバイルレイアウトの基本表示機能確認
  - **確認ポイント**: Mantineコンポーネントの条件分岐ロジックが正常動作
- 🟢 EARS要件REQ-101から確認済み

## 2. 異常系テストケース（エラーハンドリング）

### TC-101: 不正なnavigationConfigの処理
- **テスト名**: navigationConfigが不正な形式の場合適切にフォールバックされる
  - **エラーケースの概要**: 必須プロパティが欠けているnavigationConfigの処理
  - **エラー処理の重要性**: アプリケーションクラッシュを防ぎ、最低限のナビゲーション機能を維持
- **入力値**:
  ```typescript
  navigationConfig = {
    primary: [{ id: '1' }], // label, pathが欠けている
    secondary: null
  }
  ```
  - **不正な理由**: NavigationItem型の必須プロパティlabel, pathが未定義
  - **実際の発生シナリオ**: APIからの不完全なデータ受信、設定ファイルの破損
- **期待される結果**:
  - TypeScriptコンパイルエラーまたはランタイム警告
  - デフォルトナビゲーション項目の表示
  - アプリケーションクラッシュなし
  - **エラーメッセージの内容**: 「ナビゲーション設定が不正です。デフォルト設定を使用します。」
  - **システムの安全性**: Suspenseバウンダリーによる安全な代替表示
- **テストの目的**: 堅牢性とエラー耐性の確認
  - **品質保証の観点**: 本番環境での予期せぬデータエラーからの保護
- 🟡 TypeScript型システムとReactエラーハンドリングベストプラクティスから推測

### TC-102: useMediaQueryが利用できない環境での処理
- **テスト名**: useMediaQueryフックが失敗した場合の代替動作
  - **エラーケースの概要**: MediaQuery APIが利用できないブラウザ環境
  - **エラー処理の重要性**: レガシーブラウザや特殊環境でのサービス継続性
- **入力値**:
  ```typescript
  // MediaQuery APIをモック化して失敗させる
  Object.defineProperty(window, 'matchMedia', {
    value: undefined
  });
  ```
  - **不正な理由**: 古いブラウザやテスト環境でのMediaQuery API未対応
  - **実際の発生シナリオ**: IE11、一部のエンベデッド環境、サーバーサイドレンダリング
- **期待される結果**:
  - デフォルトでデスクトップレイアウト表示
  - コンソールエラーなし
  - 基本的なナビゲーション機能維持
  - **エラーメッセージの内容**: 開発者向けの警告ログ出力
  - **システムの安全性**: フォールバック戦略による安定動作
- **テストの目的**: ブラウザ互換性エラー処理の確認
  - **品質保証の観点**: 幅広い環境での動作保証
- 🟢 EDGE-001「JavaScript無効環境でも基本的なナビゲーションが機能」から確認

## 3. 境界値テストケース（最小値、最大値、null等）

### TC-201: 768px境界値でのレイアウト切り替え
- **テスト名**: 768px丁度でレイアウトが適切に切り替わる
  - **境界値の意味**: モバイル・デスクトップレイアウトの分岐点
  - **境界値での動作保証**: ブレークポイント前後での一貫した動作確認
- **入力値**:
  ```typescript
  viewportWidths = [767, 768, 769] // 境界値前後
  navigationConfig = standardConfig
  ```
  - **境界値選択の根拠**: REQ-101/102で定義された768px(48em)ブレークポイント
  - **実際の使用場面**: タブレット端末の縦横切り替え、ブラウザ画面リサイズ
- **期待される結果**:
  - 767px: モバイルレイアウト（フッター表示）
  - 768px: モバイルレイアウト（useMediaQuery(max-width: 48em)により）
  - 769px: デスクトップレイアウト（サイドバー表示）
  - **境界での正確性**: レイアウト切り替えが確実に1px単位で動作
  - **一貫した動作**: 前後の状態で要素の表示/非表示が適切
- **テストの目的**: ブレークポイント精度の確認
  - **堅牢性の確認**: 境界値付近での予期しない動作なし
- 🟢 EDGE-101「画面幅768px（ブレークポイント）で適切にレイアウトが切り替わる」から確認
