# プレイヤーノート機能 要件定義書

## 概要

ライブポーカーでの対戦相手に関する情報を効率的に管理・検索するための専用システムを構築する。プレイヤーの特徴、行動パターン、メモを体系的に記録し、復習や今後の対戦時に活用できる機能を提供する。

## 機能要件（EARS記法）

**【信頼性レベル凡例】**:
- 🔵 **青信号**: Linear要件定義・ユーザヒアリングを参考にした確実な要件
- 🟡 **黄信号**: Linear要件定義・ユーザヒアリングから妥当な推測による要件
- 🔴 **赤信号**: Linear要件定義・ユーザヒアリングにない推測による要件

### 通常要件

- REQ-001: システムはプレイヤーの基本情報（名前、作成日時、更新日時）を管理しなければならない 🔵 *Linear要件定義・ユーザヒアリング2024-09-24より*
- REQ-002: システムはプレイヤーごとにカスタマイズ可能な種別を設定できなければならない 🔵 *Linear要件定義・ユーザヒアリング2024-09-24より*
- REQ-003: システムはプレイヤーに複数のタグを割り当てることができなければならない 🔵 *Linear要件定義・ユーザヒアリング2024-09-24より*
- REQ-004: システムはプレイヤーごとにリッチテキスト形式のメモを作成・編集できなければならない 🔵 *Linear要件定義・ユーザヒアリング2024-09-24より*
- REQ-005: システムはプレイヤー名での部分一致検索機能を提供しなければならない 🔵 *Linear要件定義・ユーザヒアリング2024-09-24より*
- REQ-006: システムは既存のNote機能とは独立した新規機能として実装されなければならない 🔵 *ユーザヒアリング2024-09-24より*

### 条件付き要件

- REQ-101: プレイヤー種別を設定する場合、システムは色と名称をカスタマイズ可能にしなければならない 🔵 *Linear要件定義・ユーザヒアリング2024-09-24より*
- REQ-102: プレイヤー種別を設定する場合、システムは1プレイヤーに対して高々1つの種別のみ割り当てられなければならない 🔵 *Linear要件定義より*
- REQ-103: タグを設定する場合、システムは色と名称をカスタマイズ可能にしなければならない 🔵 *Linear要件定義・ユーザヒアリング2024-09-24より*
- REQ-104: タグを設定する場合、システムはレベル設定機能を提供しなければならない 🔵 *ユーザヒアリング2024-09-24より*
- REQ-105: タグのレベルを設定する場合、システムは色の濃淡で視覚的に強度を表現しなければならない 🔵 *ユーザヒアリング2024-09-24より*
- REQ-106: リッチテキストメモを作成する場合、システムはTipTapエディタを使用しなければならない 🔵 *ユーザヒアリング2024-09-24より*

### 状態要件

- REQ-201: プレイヤーが削除された状態では、システムは関連するすべてのタグ・メモも削除しなければならない 🟡 *既存実装パターンから妥当な推測*
- REQ-202: タグまたは種別が削除された状態では、システムは該当するプレイヤーから関連付けを解除しなければならない 🟡 *データ整合性から妥当な推測*

### オプション要件

- REQ-301: システムはプレイヤーデータのエクスポート機能を提供してもよい 🔴 *一般的なデータ管理機能からの推測*
- REQ-302: システムはタグ・種別の使用統計を表示してもよい 🔴 *管理者向け機能からの推測*

### 制約要件

- REQ-401: システムはMantineUIコンポーネントを使用しなければならない 🔵 *プロジェクト共通ルール・ユーザヒアリング2024-09-24より*
- REQ-402: システムはReact + TypeScript + Rust + Tauriの技術スタックを使用しなければならない 🔵 *プロジェクト技術制約・ユーザヒアリング2024-09-24より*
- REQ-403: システムはSQLiteデータベースを使用しなければならない 🔵 *プロジェクト技術制約・既存実装より*

## 非機能要件

### パフォーマンス

- NFR-001: プレイヤー一覧の表示は1秒以内で完了すること 🟡 *一般的なWebアプリケーション要件から妥当な推測*
- NFR-002: 検索結果の表示は500ms以内で完了すること 🟡 *ユーザビリティから妥当な推測*
- NFR-003: リッチテキストエディタの起動は300ms以内で完了すること 🟡 *TipTapエディタの一般的な性能から妥当な推測*

### ユーザビリティ

- NFR-101: プレイヤー種別とタグは色による視覚的な区別が可能であること 🔵 *Linear要件定義・ユーザヒアリング2024-09-24より*
- NFR-102: タグのレベルは色の濃淡で直感的に理解できること 🔵 *ユーザヒアリング2024-09-24より*
- NFR-103: リッチテキストエディタは中級機能（太字、斜体、色付け、箇条書き等）を提供すること 🔵 *ユーザヒアリング2024-09-24より*

### 互換性

- NFR-201: 既存のNote機能と同じブラウザ環境で動作すること 🔵 *プロジェクト技術制約より*
- NFR-202: レスポンシブレイアウト要件に準拠すること 🟡 *既存設計文書から妥当な推測*

## Edgeケース

### エラー処理

- EDGE-001: プレイヤー名が重複する場合、システムは適切なエラーメッセージを表示すること 🟡 *データ整合性から妥当な推測*
- EDGE-002: 存在しないタグ・種別を参照する場合、システムは適切にエラーハンドリングすること 🟡 *データ整合性から妥当な推測*
- EDGE-003: リッチテキストエディタでの無効なHTMLに対して適切な警告を表示すること 🔴 *TipTapエディタの一般的な課題からの推測*

### 境界値

- EDGE-101: プレイヤー名が1文字の場合でも正常に動作すること 🟡 *境界値テストから妥当な推測*
- EDGE-102: プレイヤー名が極端に長い場合（255文字）でも正常に表示されること 🟡 *境界値テストから妥当な推測*
- EDGE-103: 大量のプレイヤー（1000人以上）でもパフォーマンスが劣化しないこと 🔴 *スケーラビリティからの推測*

### データ整合性

- EDGE-201: プレイヤー削除時に孤立したタグ・種別データが残らないこと 🟡 *データ整合性から妥当な推測*
- EDGE-202: 同時編集による競合状態を適切に処理すること 🔴 *同期処理の一般的な課題からの推測*
- EDGE-203: アプリケーション強制終了時にデータの不整合が発生しないこと 🟡 *SQLiteの特性から妥当な推測*
