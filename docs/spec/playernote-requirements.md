# プレイヤーノート機能 要件定義書

## 概要

ライブポーカーでの対戦相手管理を効率化するプレイヤーノート機能の要件定義書です。復習時や後の対戦時に役立てるためのメモ管理機能を提供します。

## 機能要件（EARS記法）

**【信頼性レベル凡例】**:
- 🔵 **青信号**: Linear情報・ユーザヒアリングに基づく確実な要件
- 🟡 **黄信号**: Linear情報・ヒアリングから妥当な推測による要件
- 🔴 **赤信号**: 推測による要件（要確認）

### 通常要件

- REQ-001: システムはプレイヤーの基本情報（名前、識別子）を登録できなければならない 🔵 *ユーザヒアリング2024-09-29より*
- REQ-002: システムは同一名のプレイヤーを複数登録できなければならない 🔵 *ユーザヒアリング2024-09-29より*
- REQ-003: システムはプレイヤーにカスタマイズ可能な種別を1つ割り当てできなければならない 🔵 *Linear Project Description + ユーザヒアリングより*
- REQ-004: システムはプレイヤーにカスタマイズ可能なタグを複数割り当てできなければならない 🔵 *Linear Project Description + ユーザヒアリングより*
- REQ-005: システムはタグにレベル（I-V）を設定できなければならない 🔵 *Linear Project Description + ユーザヒアリングより*
- REQ-006: システムはタグにレベル設定を行わないオプションを提供しなければならない 🔵 *ユーザヒアリング2024-09-29より*
- REQ-007: システムはプレイヤーに簡易メモを複数作成できなければならない 🔵 *Linear Project Descriptionより*
- REQ-008: システムはプレイヤーに総合メモを1つ自動作成しなければならない 🔵 *Linear Project Descriptionより*
- REQ-009: システムはプレイヤー名と識別子での検索機能を提供しなければならない 🔵 *ユーザヒアリング2024-09-29より*
- REQ-010: システムはリスト表示とカード表示の切り替えを提供しなければならない 🔵 *ユーザヒアリング2024-09-29より*

### 条件付き要件

- REQ-101: 同一名のプレイヤーが既に存在する場合、システムは識別子入力を促すダイアログを表示しなければならない 🔵 *ユーザヒアリング2024-09-29より*
- REQ-102: プレイヤー削除時、システムは確認ダイアログを表示しなければならない 🔵 *ユーザヒアリング2024-09-29より*
- REQ-103: データエクスポート実行時、システムはJSON形式でデータを出力しなければならない 🔵 *ユーザヒアリング2024-09-29より*

### 状態要件

- REQ-201: 初期状態において、システムは「未分類」プレイヤー種別を提供しなければならない 🔵 *ユーザヒアリング2024-09-29より*
- REQ-202: 各プレイヤー作成時、システムは総合メモを自動的に作成しなければならない 🔵 *Linear Project Descriptionより*

### 制約要件

- REQ-301: システムは1プレイヤーに高々1つのプレイヤー種別のみ割り当て可能でなければならない 🔵 *Linear Project Descriptionより*
- REQ-302: システムはタグレベルを1-5（I-V）の範囲に制限しなければならない 🔵 *ユーザヒアリング2024-09-29より*
- REQ-303: システムは削除されたプレイヤーを論理削除（非表示フラグ）で処理しなければならない 🔵 *ユーザヒアリング2024-09-29より*

## 非機能要件

### パフォーマンス

- NFR-001: システムはプレイヤー一覧を1秒以内に表示しなければならない 🟡 *1000人規模対応から妥当な推測*
- NFR-002: システムは検索結果を500ms以内に表示しなければならない 🟡 *検索機能の妥当な推測*
- NFR-003: システムはリッチテキストエディタを300ms以内に起動しなければならない 🟡 *TipTapエディタの妥当な推測*

### ユーザビリティ

- NFR-101: システムはモバイルデバイスで使いやすく表示されなければならない 🔵 *ユーザヒアリング2024-09-29より（高優先度）*
- NFR-102: システムはレスポンシブデザインを採用しなければならない 🔵 *ユーザヒアリング2024-09-29より*
- NFR-103: システムはタグレベルをローマ数字と色の濃淡で表現しなければならない 🔵 *ユーザヒアリング2024-09-29より*

### セキュリティ・データ整合性

- NFR-201: システムはSQLiteを使用してローカルデータストレージを実現しなければならない 🔵 *技術スタック定義より*
- NFR-202: システムはデータの論理削除により整合性を保持しなければならない 🔵 *ユーザヒアリング2024-09-29より*

## データ仕様

### プレイヤーエンティティ

```typescript
interface Player {
  id: number;
  name: string;           // プレイヤー名
  identifier?: string;    // 識別子（同名区別用）
  player_type_id?: number; // プレイヤー種別ID（nullable）
  is_deleted: boolean;    // 論理削除フラグ
  created_at: string;
  updated_at: string;
}
```

### プレイヤー種別エンティティ

```typescript
interface PlayerType {
  id: number;
  name: string;           // 種別名（強さレベル分類）
  color: string;          // HEXカラーコード
  created_at: string;
}
```

### タグエンティティ

```typescript
interface Tag {
  id: number;
  name: string;           // タグ名
  color: string;          // HEXカラーコード
  has_level: boolean;     // レベル設定有無
  created_at: string;
}
```

### プレイヤータグ関連エンティティ

```typescript
interface PlayerTag {
  id: number;
  player_id: number;
  tag_id: number;
  level?: number;         // 1-5 (I-V)、has_level=falseの場合null
  created_at: string;
}
```

### メモエンティティ

```typescript
interface PlayerNote {
  id: number;
  player_id: number;
  title?: string;         // 簡易メモのタイトル
  content: string;        // HTML形式
  note_type: 'simple' | 'comprehensive'; // 簡易 or 総合
  created_at: string;
  updated_at: string;
}
```

## 用語定義

- **プレイヤー**: 対戦相手として登録される人物
- **識別子**: 同名プレイヤーを区別するための短いテキスト
- **プレイヤー種別**: 強さレベル分類（初心者、中級者、上級者等）
- **タグ**: プレイヤーの特徴を表すラベル（レベル付き可能）
- **簡易メモ**: ハンド履歴など短期的なメモ
- **総合メモ**: プレイヤー分析など包括的なメモ
- **論理削除**: データを物理的に削除せず、フラグで非表示にする手法
