# プレイヤーノート機能 受け入れ基準

## 概要

このドキュメントはプレイヤーノート機能の受け入れ基準とテスト項目を記載します。新規機能として独立して開発される前提での品質基準を定義しています。

## 機能テスト基準

**【信頼性レベル凡例】**:
- 🔵 **青信号**: Linear要件定義・ユーザヒアリングを参考にした確実なテスト基準
- 🟡 **黄信号**: Linear要件定義・ユーザヒアリングから妥当な推測によるテスト基準
- 🔴 **赤信号**: Linear要件定義・ユーザヒアリングにない推測によるテスト基準

### REQ-001: プレイヤー基本情報管理の受け入れ基準 🔵 *Linear要件定義・ユーザヒアリング2024-09-24より*

**Given（前提条件）**:
- システムが正常に起動している
- データベースが初期化されている

**When（実行条件）**:
- 新規プレイヤーの名前「田中太郎」を入力する
- プレイヤー登録ボタンをクリックする

**Then（期待結果）**:
- プレイヤーが正常に登録される
- 作成日時・更新日時が自動設定される
- プレイヤー一覧に表示される

**テストケース**:
- [ ] 正常系: 有効な名前でプレイヤーを登録できる
- [ ] 異常系: 空白名前での登録がエラーになる
- [ ] 異常系: 重複名前での登録がエラーになる
- [ ] 境界値: 1文字の名前で登録できる
- [ ] 境界値: 255文字の名前で登録できる

### REQ-002: カスタマイズ可能なプレイヤー種別の受け入れ基準 🔵 *Linear要件定義・ユーザヒアリング2024-09-24より*

**Given（前提条件）**:
- システムが起動している
- プレイヤーが1人以上登録されている

**When（実行条件）**:
- 種別作成画面で名称「アグレッシブ」、色「#FF0000」を設定する
- 作成ボタンをクリックする
- プレイヤーに作成した種別を割り当てる

**Then（期待結果）**:
- 種別が正常に作成される
- プレイヤーに種別が設定される
- 設定した色で種別が表示される

**テストケース**:
- [ ] 正常系: カスタム種別を作成・割り当てできる
- [ ] 制約系: 1プレイヤーに複数種別は割り当てできない
- [ ] 異常系: 重複名称の種別は作成できない
- [ ] UI系: 色選択UIが正常に動作する

### REQ-003: 複数タグ割り当ての受け入れ基準 🔵 *Linear要件定義・ユーザヒアリング2024-09-24より*

**Given（前提条件）**:
- プレイヤーが登録されている
- タグ「ブラフ好き」「ルーズ」が作成されている

**When（実行条件）**:
- プレイヤー詳細画面でタグ割り当てを実行する
- 「ブラフ好き」と「ルーズ」を同時に選択する
- 割り当てボタンをクリックする

**Then（期待結果）**:
- 両方のタグが同時に割り当てられる
- プレイヤー一覧で両タグが表示される
- タグの個別削除が可能である

**テストケース**:
- [ ] 正常系: 複数タグの同時割り当てができる
- [ ] 正常系: 同一タグの複数回割り当てができる
- [ ] 正常系: タグの個別削除ができる
- [ ] 境界値: 大量タグ（50個以上）の割り当てができる

### REQ-104: タグレベル設定の受け入れ基準 🔵 *ユーザヒアリング2024-09-24より*

**テストケース**:
- [ ] 正常系: レベル1-10の設定ができる
- [ ] 正常系: 色の濃淡がレベルに連動する
- [ ] 正常系: レベル情報が永続化される
- [ ] UI系: レベル調整UIが直感的に操作できる

### REQ-004: リッチテキストメモの受け入れ基準 🔵 *Linear要件定義・ユーザヒアリング2024-09-24より*

**テストケース**:
- [ ] 正常系: 基本フォーマット（太字、斜体、下線）が使用できる
- [ ] 正常系: 色付け機能が正常に動作する
- [ ] 正常系: 箇条書き・番号付きリストが作成できる
- [ ] 正常系: メモの編集・更新ができる
- [ ] 異常系: 無効なHTMLタグが適切に処理される

- [ ] 正常系: リアルタイム検索が正常に動作する

## 非機能テスト基準

### パフォーマンステスト

**NFR-001: プレイヤー一覧表示パフォーマンス 🟡 *一般的なWebアプリケーション要件から妥当な推測***

- [ ] 初期表示: 100プレイヤーの一覧が1秒以内で表示される
- [ ] スクロール: 仮想スクロールで1000プレイヤーでも滑らかに動作する
- [ ] フィルタリング: 検索結果の表示が500ms以内で完了する
- [ ] データ読み込み: SQLiteからのデータ取得が200ms以内で完了する

**テスト方法**:
- パフォーマンス測定ツール: Vitestのperformance API
- 大量データ生成: テスト用の1000プレイヤーデータを自動生成
- 測定項目: 初期表示時間、検索応答時間、メモリ使用量

- [ ] レベル表現: タグレベル1-10が色の濃淡で識別できる
- [ ] アクセシビリティ: 色覚障害者でも形状・パターンで区別できる

### データ整合性テスト

**EDGE-201: カスケード削除テスト 🟡 *データ整合性から妥当な推測***

**テストシナリオ**:
- プレイヤーに種別・タグ・メモが関連付けられている状況
- プレイヤーを削除実行
- 関連データの削除確認

**合格基準**:
- [ ] プレイヤー削除時に関連するメモがすべて削除される
- [ ] プレイヤー削除時にタグ関連付けがすべて削除される
- [ ] 種別・タグマスタは削除されない（他のプレイヤーが使用中でも）
- [ ] データベース制約違反が発生しない

**EDGE-202: 同時編集競合テスト 🔴 *同期処理の一般的な課題からの推測***

**テストシナリオ**:
- 同一プレイヤーのメモを複数ウィンドウで同時編集
- 片方のウィンドウで保存実行
- もう片方のウィンドウで保存実行

**合格基準**:
- [ ] 後勝ち（Last Write Wins）で競合が解決される
- [ ] データ破損が発生しない
- [ ] 適切な警告メッセージが表示される
- [ ] ユーザーに競合状況が通知される
