# TDD 完全テスト実装ルール

## 目的

TDD開発において、テストケース定義書で計画した**全てのテストケースを必ず実装する**ことを義務付け、品質基準を確保します。

## 基本原則

### ✅ 必須事項

1. **全テストケース実装の義務化**
   - `tdd-testcases` フェーズで定義した全テストケースを **100%実装** すること
   - 部分的な実装で終わらせないこと
   - テストケース定義書に記載された件数と実装件数が一致すること

2. **実装完了の基準**
   - 正常系テストケース: 100%実装
   - 異常系テストケース: 100%実装
   - 境界値テストケース: 100%実装
   - 特殊ケーステストケース: 100%実装
   - **全体実装率: 100%必須**

3. **品質検証の基準**
   - `/tdd-verify-complete` で実装率が100%であることを確認
   - 実装率が100%未満の場合は、残りのテストケースを実装してから次のステップへ進む

## 実装プロセス

### Green Phase（tdd-green）での対応

#### 最初のGreen Phase
1. **必須実装**: 正常系の基本的なテストケース（2-3件）
2. **目的**: 基本機能が動作することを確認

#### 追加のGreen Phase（必要に応じて繰り返し）
1. **全テストケースの段階的実装**
   - 未実装のテストケースを順次実装
   - 各実装後にテストが成功することを確認

2. **実装順序の推奨**
   - 正常系 → 異常系 → 境界値 → 特殊ケース
   - または、機能別にグループ化して実装

### Verify Complete Phase（tdd-verify-complete）での確認

1. **実装率チェック**
   - 予定テストケース数と実装テストケース数を比較
   - 実装率が100%であることを確認

2. **未実装の場合の対応**
   ```
   実装率 < 100% の場合:
   → tdd-green に戻って残りのテストケースを実装
   → 全テストケース実装完了まで繰り返し
   ```

3. **完了基準**
   ```
   ✅ 実装率: 100% (全テストケース実装済み)
   ✅ 成功率: 100% (全テスト成功)
   ```

## 禁止事項

### ❌ 部分実装での次ステップ移行

- **禁止**: テストケースの一部のみ実装して次のタスクに進むこと
- **理由**: 品質基準未達、機能の不完全な実装につながる

### ❌ テストケースの削除・省略

- **禁止**: 実装が困難という理由でテストケースを削除・省略すること
- **対応**: 実装が困難な場合は設計を見直すか、要件を再確認する

### ❌ 実装率80%での妥協

- **禁止**: 「80%実装したから十分」と判断して次に進むこと
- **要求**: 100%実装が必須

## 例外規定

以下の場合のみ、テストケースの未実装が許容されます：

1. **要件変更による不要化**
   - 要件定義が変更され、テストケースが不要になった場合
   - 変更理由をドキュメントに明記すること

2. **技術的制約による延期**
   - 外部依存関係の不備など、技術的な理由で実装不可能な場合
   - 代替手段を検討し、Issue化して追跡すること

## 実装例

### 良い例 ✅

```
テストケース定義:
- 正常系: 8件
- 異常系: 8件
- 境界値: 4件
- 特殊ケース: 2件
合計: 22件

実装状況:
- 正常系: 8件 ✅
- 異常系: 8件 ✅
- 境界値: 4件 ✅
- 特殊ケース: 2件 ✅
合計: 22件 ✅

実装率: 100% ✅
→ 次のステップへ進む
```

### 悪い例 ❌

```
テストケース定義:
- 正常系: 8件
- 異常系: 8件
- 境界値: 4件
- 特殊ケース: 2件
合計: 22件

実装状況:
- 正常系: 5件 ⚠️
- 異常系: 5件 ⚠️
- 境界値: 2件 ⚠️
- 特殊ケース: 2件 ✅
合計: 14件 ❌

実装率: 63.6% ❌
→ 残り8件を実装してから次のステップへ
```

## 品質保証の効果

### 完全実装のメリット

1. **バグ検出率の向上**
   - 異常系・境界値を網羅することで、潜在的なバグを事前に発見

2. **リファクタリングの安全性**
   - 包括的なテストにより、リファクタリング時の品質を保証

3. **仕様の明確化**
   - 全ケースのテストにより、機能仕様が明確になる

4. **メンテナンス性の向上**
   - 将来の変更時にテストが安全網として機能

## まとめ

- **TDD開発では全テストケースを100%実装すること**
- **部分実装での次ステップ移行は禁止**
- **tdd-verify-completeで100%実装を確認**
- **例外は要件変更または技術的制約のみ**

---

このルールにより、TDD開発の品質基準を確保し、高品質なソフトウェアを開発します。
