# TDDコマンド セッション管理ルール

## コンテキスト管理の重要性

TDD開発フローでは、各ステップ（Red → Green → Refactor）を実行するたびに多くの情報がコンテキストに蓄積されます。効率的な開発を継続するために、適切なタイミングでコンテキストを最適化する必要があります。

## ステップ終了時のCompactルール

### 基本方針

TDDタスクを実装する際は、**各ステップを終了するごとに `/compact` コマンドを実行すること**

### 実行タイミング

以下のコマンド完了時に `/compact` を実行：

1. **tdd-red 完了後**
   - 失敗するテストの作成が完了
   - GitHub Issueへのコメント追加完了
   - 自動コミット完了

2. **tdd-green 完了後**
   - テストを通す最小実装が完了
   - GitHub Issueへのコメント追加完了
   - 自動コミット完了

3. **tdd-refactor 完了後**
   - リファクタリングが完了
   - GitHub Issueへのコメント追加完了
   - 自動コミット完了

### Compact時の保持情報

`/compact` 実行時には、以下の情報を**必ず保持**すること：

#### 1. 実行したコマンド履歴
- どのTDDコマンドを実行したか
- 引数（Issue番号等）
- 実行順序

#### 2. プロジェクトルール
- `docs/rule/` ディレクトリ内の全ルール
- 技術スタック定義（`docs/tech-stack.md`）
- プロジェクト固有のガイドライン

#### 3. 開発コンテキスト
- 現在作業中のIssue番号
- 実装対象の機能概要
- 次に実行すべきステップ

#### 4. 重要な実装決定事項
- 採用した設計パターン
- 使用しているライブラリ・フレームワーク
- 特殊な実装理由やトレードオフ

### Compact実行例

```bash
# tdd-red完了後
/compact

# compactプロンプト例：
# 「Issue #125のRed Phase完了。失敗テスト10件作成。
#  次はGreen Phase（/tdd-green 125）を実行予定。
#  技術スタック: Rust + Tauri + React
#  実装対象: プレイヤー種別CRUD機能
#  プロジェクトルール: docs/rule/配下の全ルール適用」
```

### Compactのメリット

1. **コンテキストサイズの最適化**
   - トークン使用量の削減
   - レスポンス速度の向上

2. **重要情報のフォーカス**
   - 不要な詳細を削除
   - 必要な情報のみを保持

3. **連続開発の効率化**
   - 次のステップにスムーズに移行
   - 開発フローの中断を防止

4. **コスト削減**
   - API利用コストの最適化
   - 長時間セッションでも効率的

## 注意事項

### Compactしてはいけないタイミング

- デバッグ中で詳細なエラー情報が必要な時
- 複雑な問題解決のために過去のコンテキストが必要な時
- レビューや振り返りのために全履歴が必要な時

### Compact失敗時の対処

もし `/compact` の実行が適切でなかった場合：
1. `/clear` で完全にクリアして再開
2. 必要な情報をGitHub Issueコメントから再取得
3. TDDコマンドを再実行

## ワークフロー例

### 標準的なTDD開発フロー

```bash
# 1. 要件定義
/tdd-requirements 125
# → GitHub Issueコメント確認
# → /compact（必要に応じて）

# 2. テストケース作成
/tdd-testcases 125
# → GitHub Issueコメント確認
# → /compact（必要に応じて）

# 3. Red Phase
/tdd-red 125
# → 失敗テスト実装
# → 自動コミット
# → /compact ← ここで必ず実行

# 4. Green Phase
/tdd-green 125
# → 最小実装
# → 自動コミット
# → /compact ← ここで必ず実行

# 5. Refactor Phase
/tdd-refactor 125
# → リファクタリング
# → 自動コミット
# → /compact ← ここで必ず実行

# 6. 品質確認
/tdd-verify-complete 125
# → 完了
```

## セッション管理のベストプラクティス

1. **定期的なCompact**
   - 各TDDステップ完了後に実行
   - 長時間作業時は適宜実行

2. **重要情報の記録**
   - GitHub Issueコメントに重要な決定事項を記録
   - ローカルメモは最小限に

3. **コマンド履歴の明確化**
   - 実行したコマンドを記録
   - 次のステップを明示

4. **コンテキスト再構築の準備**
   - 必要な情報はGitHub上に保存
   - いつでも再開できる状態を維持
