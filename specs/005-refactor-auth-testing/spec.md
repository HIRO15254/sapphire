## Clarifications

### Session 2025-12-02

- Q: OAuth認証とメール/パスワード認証で同じメールアドレスを使用した場合、アカウントは統合されるか、別々に作成されるか？ → A: アカウントリンク方式を採用。同一メールアドレスは同一ユーザーとして扱い、複数の認証方法（OAuth、Credentials）をリンクする

## User Scenarios & Testing *(mandatory)*

### User Story 1 - メールアドレス/パスワードでのユーザー登録・ログイン (Priority: P1)

ユーザーは、OAuthプロバイダー（Google/GitHub）を持っていなくても、メールアドレスとパスワードを使ってアカウントを作成し、アプリケーションにログインできる。これにより、OAuthに依存せずにアプリケーションを利用でき、E2Eテストの実行が可能になる。

**Why this priority**: E2Eテストを実行するために必須の機能であり、他のすべてのテスト・品質向上作業の基盤となる。OAuthプロバイダーではテスト自動化が困難なため、メール/パスワード認証が不可欠。

**Independent Test**: メールアドレスとパスワードで新規登録し、ログアウト後に同じ認証情報でログインし、セッション管理画面にアクセスできることで価値を検証できる。

**Acceptance Scenarios**:

1. **Given** 未登録のユーザーがログインページを開いている、**When** 「アカウント作成」リンクをクリックしてメールアドレスとパスワードを入力する、**Then** 新しいアカウントが作成され、ログイン状態でホーム画面に遷移する
2. **Given** 登録済みのユーザーがログインページを開いている、**When** 正しいメールアドレスとパスワードを入力する、**Then** ログイン状態でホーム画面に遷移する
3. **Given** ユーザーがログインページを開いている、**When** 間違ったパスワードを入力する、**Then** エラーメッセージが表示され、ログインに失敗する
4. **Given** 既に登録済みのメールアドレスで登録を試みる、**When** 登録フォームを送信する、**Then** 「このメールアドレスは既に登録されています」というエラーメッセージが表示される
5. **Given** ユーザーがログイン状態である、**When** ログアウトボタンをクリックする、**Then** セッションが終了し、ログインページにリダイレクトされる

---

### User Story 2 - E2Eテストによる主要機能の自動検証 (Priority: P2)

開発者は、メールアドレス/パスワード認証を利用したE2Eテストを実行し、アプリケーションの主要機能（ログイン、セッション作成、編集、削除、フィルタリング）が正しく動作することを自動的に検証できる。

**Why this priority**: 品質保証のためにE2Eテストは必須であり、メール/パスワード認証の実装直後に実行可能になる。これにより、リグレッションを防止し、継続的な品質向上が可能になる。

**Independent Test**: テストスイートを実行し、すべてのテストがパスすることで、主要機能の動作が保証される。

**Acceptance Scenarios**:

1. **Given** テスト環境が準備されている、**When** E2Eテストスイートを実行する、**Then** テストユーザーが自動作成されログインが成功する
2. **Given** テストユーザーでログインしている、**When** 新規セッション作成のE2Eテストを実行する、**Then** セッションが正しく作成され一覧に表示される
3. **Given** テストセッションが存在する、**When** セッション編集のE2Eテストを実行する、**Then** 変更内容が正しく保存される
4. **Given** テストセッションが存在する、**When** セッション削除のE2Eテストを実行する、**Then** セッションが削除され一覧から消える
5. **Given** 複数のテストセッションが存在する、**When** フィルタリング機能のE2Eテストを実行する、**Then** 条件に一致するセッションのみが表示される

---

### User Story 3 - ドキュメントの整備と実装状況の同期 (Priority: P3)

新規開発者やプロジェクト関係者は、README.mdをはじめとした各種ドキュメントを読み、アプリケーションの現在の実装状況、機能一覧、セットアップ手順を正確に把握できる。

**Why this priority**: ドキュメントが実装と乖離していると、新規参加者のオンボーディングが困難になり、プロジェクトのメンテナビリティが低下する。現状のREADME.mdは「実装予定」と記載されているが、実際には多くの機能が実装済みであるため更新が必要。

**Independent Test**: README.mdおよび関連ドキュメントを読んで、記載内容と実際の実装が一致していることを目視確認できる。

**Acceptance Scenarios**:

1. **Given** README.mdを開く、**When** 「主な機能」セクションを確認する、**Then** 実装済みの機能がすべて記載されており、「実装予定」ではなく「実装済み」として記載されている
2. **Given** README.mdを開く、**When** セットアップ手順に従う、**Then** 開発環境が正常に起動し、アプリケーションを利用できる
3. **Given** ドキュメントを確認する、**When** メール/パスワード認証の設定手順を探す、**Then** 新しく追加された認証方法のセットアップ手順が記載されている
4. **Given** ドキュメントを確認する、**When** テスト実行手順を探す、**Then** E2Eテストの実行方法が記載されている

---

### User Story 4 - フロントエンドUIのリデザイン (Priority: P4)

ユーザーは、より洗練された統一感のあるUIを通じてアプリケーションを利用でき、機能は維持したまま視覚的な品質と使いやすさが向上する。

**Why this priority**: 既存機能を維持しながらのリデザインは、機能追加やテスト整備より優先度が低い。ただし、ユーザー体験の向上とプロダクト品質に寄与する重要な作業。

**Independent Test**: 各画面を操作して、デザインの一貫性、レスポンシブ対応、視認性の向上を確認できる。

**Acceptance Scenarios**:

1. **Given** アプリケーションを開く、**When** ホーム画面を表示する、**Then** ログイン前後で一貫したデザインが適用されている
2. **Given** セッション一覧画面を開く、**When** 統計・フィルター・リストを確認する、**Then** 各コンポーネントが視覚的に統一されたデザインで表示される
3. **Given** モバイルデバイスでアクセスする、**When** 各画面を表示する、**Then** レスポンシブデザインが適用され、使いやすいレイアウトになっている
4. **Given** セッション作成モーダルを開く、**When** フォームを操作する、**Then** 入力フィールドのデザインが統一され、操作しやすい

---

### User Story 5 - コンポーネントテストの追加 (Priority: P5)

開発者は、Presentationコンポーネントの単体テストを実行し、各UIコンポーネントが期待通りに描画・動作することを自動的に検証できる。

**Why this priority**: E2Eテストだけでなく、個別コンポーネントの品質を保証するためにコンポーネントテストも必要。ただし、E2Eテストが優先されるため、後続の作業として実施。

**Independent Test**: コンポーネントテストスイートを実行し、すべてのテストがパスすることで、UIコンポーネントの動作が保証される。

**Acceptance Scenarios**:

1. **Given** テスト環境が準備されている、**When** コンポーネントテストスイートを実行する、**Then** 主要なPresentationコンポーネントのテストがパスする
2. **Given** SessionCardコンポーネントのテストを確認する、**When** 異なるpropsでレンダリングする、**Then** 勝ち/負け/引き分けセッションが正しく表示される
3. **Given** SessionFormコンポーネントのテストを確認する、**When** バリデーションエラーを発生させる、**Then** エラーメッセージが正しく表示される

---

### Edge Cases

- パスワードの強度要件を満たさない場合、どのようなエラーメッセージが表示されるか？
- メールアドレスの形式が不正な場合、どのようなバリデーションが行われるか？
- E2Eテスト実行中にネットワークエラーが発生した場合、テストは適切にリトライまたはフェイルするか？
- E2Eテストで作成されたテストデータは、テスト終了後に適切にクリーンアップされるか？
- OAuth認証とメール/パスワード認証で同じメールアドレスを使用した場合 → アカウントリンク方式により同一ユーザーとして扱われ、両方の認証方法でログイン可能
- UIリデザイン後も、すべての既存機能（モーダル、フィルター、タグ選択、リッチテキスト編集など）が正常に動作するか？

## Requirements *(mandatory)*

### Functional Requirements

**メールアドレス/パスワード認証**:
- **FR-001**: システムは、メールアドレスとパスワードを使用したユーザー登録機能を提供しなければならない
- **FR-002**: システムは、メールアドレスとパスワードを使用したログイン機能を提供しなければならない
- **FR-003**: システムは、パスワードを安全にハッシュ化して保存しなければならない
- **FR-004**: システムは、メールアドレスの形式バリデーションを行わなければならない
- **FR-005**: システムは、パスワードの強度要件（最低8文字）を検証しなければならない
- **FR-006**: システムは、既存のOAuth認証（Google/GitHub）と並行してメール/パスワード認証をサポートしなければならない
- **FR-007**: システムは、同一メールアドレスでの重複登録を防止しなければならない
- **FR-007a**: システムは、同一メールアドレスに対して複数の認証方法（OAuth、Credentials）をリンクし、単一ユーザーアカウントとして管理しなければならない（アカウントリンク方式）
- **FR-008**: システムは、ログイン失敗時に適切なエラーメッセージを表示しなければならない（セキュリティ上、具体的な失敗理由は開示しない）

**E2Eテスト**:
- **FR-009**: システムは、Playwrightを使用したE2Eテストをサポートしなければならない
- **FR-010**: E2Eテストは、テスト用の認証情報を使用して自動ログインできなければならない
- **FR-011**: E2Eテストは、セッションのCRUD操作（作成・読取・更新・削除）をカバーしなければならない
- **FR-012**: E2Eテストは、フィルタリング機能をカバーしなければならない
- **FR-013**: E2Eテストは、テストデータの作成・クリーンアップを自動的に行わなければならない

**ドキュメント更新**:
- **FR-014**: README.mdは、実装済み機能の一覧を正確に反映しなければならない
- **FR-015**: README.mdは、メール/パスワード認証のセットアップ手順を含まなければならない
- **FR-016**: README.mdは、E2Eテストの実行方法を含まなければならない
- **FR-017**: 関連ドキュメントは、現在の実装状況と同期しなければならない

**UIリデザイン**:
- **FR-018**: UIリデザインは、既存の全機能を維持しなければならない（機能削減なし）
- **FR-019**: UIは、デスクトップ・タブレット・モバイルでレスポンシブに動作しなければならない
- **FR-020**: UIは、Mantineのデザインシステムに準拠した一貫したスタイルを持たなければならない
- **FR-021**: UIは、WCAG 2.1 AAのアクセシビリティ基準を満たさなければならない

**コンポーネントテスト**:
- **FR-022**: 主要なPresentationコンポーネントは、単体テストを持たなければならない
- **FR-023**: コンポーネントテストは、Vitest + React Testing Libraryを使用しなければならない
- **FR-024**: コンポーネントテストは、異なる状態（正常・エラー・空状態等）をカバーしなければならない

### Key Entities

- **ユーザー資格情報 (Credentials)**: メールアドレス/パスワード認証のための追加属性。既存のユーザーエンティティにパスワードハッシュフィールドを追加。OAuthユーザーはパスワードがnull。

- **E2Eテストフィクスチャ (Test Fixtures)**: E2Eテストで使用されるテストユーザーとテストデータの定義。テスト実行前に作成され、実行後にクリーンアップされる。

## Success Criteria *(mandatory)*

### Measurable Outcomes

**認証機能**:
- **SC-001**: ユーザーは30秒以内にメールアドレスとパスワードで新規登録を完了できる
- **SC-002**: ユーザーは10秒以内にメールアドレスとパスワードでログインできる
- **SC-003**: 認証エラーは100%適切なエラーメッセージとともにユーザーに通知される

**E2Eテスト**:
- **SC-004**: E2Eテストスイートは5分以内に完了する
- **SC-005**: E2Eテストは主要機能（認証、セッションCRUD、フィルタリング）の80%以上をカバーする
- **SC-006**: E2Eテストは、10回連続実行しても安定してパスする（flaky testなし）

**ドキュメント**:
- **SC-007**: README.mdの内容と実際の実装の一致率が100%である
- **SC-008**: 新規開発者はドキュメントのみで15分以内に開発環境をセットアップできる

**UIリデザイン**:
- **SC-009**: 既存の全機能（セッションCRUD、フィルタリング、タグ、場所管理、リッチテキスト編集）がリデザイン後も100%動作する
- **SC-010**: モバイルデバイス（幅375px以上）で全画面が正しく表示され操作可能である
- **SC-011**: 色のコントラスト比がWCAG 2.1 AAの基準（4.5:1以上）を満たす

**コンポーネントテスト**:
- **SC-012**: 主要Presentationコンポーネント（SessionCard, SessionForm, SessionList, SessionStats, SessionFilters）のテストカバレッジが80%以上である
- **SC-013**: コンポーネントテストスイートは30秒以内に完了する
